# Cyber-Vault Chrome Extension

Cyberpunk-themed password manager Chrome extension. Frontend-only (no background service worker). Uses Convex as the cloud database backend. All credentials are encrypted client-side before being sent to Convex — the server never sees plaintext.

## How It Works

### Authentication
- No user accounts. A single **master password** is used to derive the AES-256-GCM encryption key via PBKDF2 (100,000 iterations, SHA-256).
- On first run, a `SYSTEM_CANARY` record is created: the string `"CANARY_OK"` encrypted with the master password and stored in Convex.
- On every login, the canary is fetched and decrypted. If it decrypts to `"CANARY_OK"`, the password is correct.
- The master password is **never stored anywhere**.

### Encryption
- Algorithm: **AES-256-GCM** via Web Crypto API (`crypto.subtle`)
- Key derivation: **PBKDF2** — random 16-byte salt, 100,000 iterations, SHA-256
- Each entry has its own random `salt` (16 bytes) and `iv` (12 bytes), stored as Base64 in Convex
- Username (`u`) and password (`p`) are encrypted separately but share the same salt/iv per entry
- Extra fields (`fields`) are stored as an encrypted JSON string

### Auto-Lock
- Configured in Settings (minutes). `0` = disabled.
- `unlockTime` is persisted to `chrome.storage.local` under `sessionData` so it survives popup close/reopen.
- On popup open, elapsed time is checked against the autolock setting. If expired, the session is cleared and the user must re-enter the master password.
- Activity (click, keypress, mousemove) resets the timer and updates `chrome.storage.local`.

### State (app.js globals)
| Variable | Type | Purpose |
|---|---|---|
| `masterKey` | `string` | Master password in memory (lost on popup close — by design) |
| `unlockTime` | `number` | Timestamp of last activity (persisted to storage) |
| `vaultEntries` | `array` | Decrypted entries loaded into memory after login |
| `currentGroup` | `string` | Currently selected domain filter |
| `settings` | `object` | `{ autolock: number, autodomain: boolean }` |

## Convex Backend

### Table: `vault`
| Field | Type | Notes |
|---|---|---|
| `domain` | string | Plaintext — used for grouping/filtering |
| `salt` | string | Base64 — shared salt for u+p encryption |
| `iv` | string | Base64 — shared IV for u+p encryption |
| `u` | string | Encrypted username |
| `p` | string | Encrypted password |
| `fields` | string | Encrypted JSON of extra key-value fields |
| `createdAt` | number | Unix timestamp |

### Functions (`convex/passwords.ts`)
- `passwords:list` — returns all vault records
- `passwords:getCanary` — returns the `SYSTEM_CANARY` record
- `passwords:add` — inserts a new record
- `passwords:update` — patches an existing record by ID
- `passwords:remove` — deletes a record by ID

## UI Structure (index.html)

All views are in one HTML file, toggled with the `.hidden` class.

- `#login-container` — master password entry screen
- `#vault-container` — main vault view
  - Sidebar: domain group list + search input
  - Content: entry cards with header (domain, user, action buttons) + collapsible details panel
- `#add-modal` — new credential form with password generator
- `#edit-modal` — edit existing credential
- `#settings-modal` — autolock timer + auto-domain search toggle

## Build & Load

### Setup
```bash
# Install deps
npm install

# Start Convex backend (first time or after schema changes)
npx convex dev

# Build extension
npm run build
```

### Load in Chrome
1. Go to `chrome://extensions`
2. Enable **Developer mode**
3. Click **Load unpacked**
4. Select the `dist/` folder (NOT the project root)

> You must always load `dist/` because `app.js` uses `import.meta.env` and bare module imports (`convex/browser`) that only work after Vite bundles them.

### After Code Changes
```bash
npm run build
```
Then click the **reload icon** on the extension card in `chrome://extensions`.

## Known Quirks

- **Popup lifecycle**: Chrome destroys the popup JS context every time the popup closes. `masterKey` is intentionally not persisted (security). Only `unlockTime` is persisted so the autolock timer works across opens.
- **`fields` encryption**: Extra fields share the same salt/iv as u/p per entry. They are stored as a JSON string and encrypted as one blob.
- **Domain is plaintext**: The `domain` field is stored unencrypted in Convex to allow server-side grouping/filtering without decrypting everything.
